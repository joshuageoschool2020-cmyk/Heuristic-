# System Patterns – Architecture & Standards

## Architecture Overview

### Layered Design
```
Frontend (HTML/JS)
    ↓ (fetch API calls)
API Layer (FastAPI in api.py)
    ↓ (function call)
Analysis Engine (brain.py)
    ↓ (imports as needed)
Support Modules (cognitive.py, network.py, passport.py, telemetry.py)
```

### Module Responsibilities

| Module | Role | Status |
|--------|------|--------|
| `api.py` | FastAPI app, request routing, UI serving | Production |
| `brain.py` | Core `process_concept()` analysis logic | In Development |
| `cognitive.py` | Heuristic reasoning utilities | Support |
| `network.py` | Network analysis and connectivity | Support |
| `passport.py` | Authentication/verification framework | Support |
| `telemetry.py` | Logging and metrics collection | Support |
| `index.html` | Legacy HTML (superseded by UI_HTML in api.py) | Deprecated |

## API Design Principles

1. **Contract-First**: All endpoints define Pydantic request/response models upfront
2. **Stateless**: No session state; each request self-contained
3. **JSON-First**: All responses return valid JSON (no custom serialization)
4. **Error Explicit**: HTTP status codes and error payloads clearly indicate failure mode

## Data Contracts

### POST `/explain` Contract
**Request**:
```python
{
    "text": str  # Input sequence to analyze
}
```

**Response (200 OK)**:
```python
{
    "result": str,      # Analysis output text
    "confidence": float # Confidence score 0-100
}
```

**Response (400 Bad Request)**:
```python
{
    "detail": str  # Error message (auto-generated by FastAPI)
}
```

## Logging Standards

### Required Logging Points
- [ ] API request received (endpoint, timestamp, input length)
- [ ] Analysis started (input hash for traceability)
- [ ] Analysis completed (processing time, confidence score)
- [ ] API response sent (HTTP status, response size)
- [ ] Errors caught (exception type, stack trace)

### Logging Format
```python
import logging
logger = logging.getLogger(__name__)
logger.info(f"Processing text sequence: {len(text)} chars, confidence: {score}")
```

## Deployment Standards

### Environment Variables
- `PORT` (default: 10000) – Server port
- `RENDER_GIT_COMMIT` (auto-set by Render) – Deployment version
- Future: `TIER_LIMIT` (API rate limit), `LOG_LEVEL`, `DB_URL`

### Render Configuration
- Python 3.x runtime
- Build command: `pip install -r requirements.txt`
- Start command: `python api.py`
- Health check: GET `/` should return 200 with HTML

## Testing Patterns

### Local Testing
```bash
# Start server
python api.py

# Test GET /
curl http://localhost:10000/

# Test POST /explain
curl -X POST http://localhost:10000/explain \
  -H "Content-Type: application/json" \
  -d '{"text": "sample analysis"}'
```

### Production Testing (on Render)
- Same curl commands against deployed URL
- Monitor response times and error rates
- Check server logs for exceptions

## Dependency Management
- `requirements.txt` pins all Python versions for reproducibility
- No external API dependencies (all processing local)
- Future: Redis for caching, PostgreSQL for analytics
